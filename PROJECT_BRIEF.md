# vaults.fyi Research Engine — Project Brief

## Overview

Build a programmatic content publishing pipeline that generates data-driven research pages using the vaults.fyi API. These pages should rank for DeFi yield queries on traditional search (SEO) and be cited by AI answer engines including ChatGPT, Perplexity, and similar tools (AEO).

The strategic goal: position vaults.fyi as **the canonical data source for DeFi yield markets**.

Target queries:
- "best USDC yields DeFi"
- "Morpho USDC APY"
- "ETH staking yield"
- "DeFi lending rates today"

## Architecture

### Stack

- **Astro** — static site generator. Purpose-built for content/SEO. Excellent meta tag control, built-in sitemap, minimal JS output, fast page loads.
- **Node.js build script** — fetches API data at build time, passes to Astro templates
- **GitHub Actions** — daily cron job triggers rebuild + redeploy
- **Hosting** — see hosting decision below

### Data flow

```
GitHub Actions (daily cron: 06:00 UTC)
        │
        ▼
  Node.js fetch script
  ├── GET /v2/detailed-vaults  (filtered per page)
  ├── GET /v2/historical-benchmarks/{network}
  └── GET /v2/benchmarks/{network}
        │
        ▼
  Write JSON data files to src/data/
        │
        ▼
  Astro build (static HTML generation)
        │
        ▼
  Deploy to host
        │
        ▼
  blog.vaults.fyi/research/*
```

### Key API endpoints used

All APY values are raw decimals — multiply by 100 to display as percentages.

**Vault discovery** (`/v2/detailed-vaults`)
- Filter params: `assets`, `protocols`, `networks`, `minTvl`
- Returns: `apy`, `apy1d`, `apy7d`, `apy30d`, `tvl`, `reputationScore`, `flags`, `capacity`, `rewards`
- Use `apy7d` or `apy30d` for displayed rates (smoother, more representative than spot APY)
- Sort by `apy7d` descending for "best yields" pages

**Benchmarks** (`/v2/benchmarks/{network}`, `/v2/historical-benchmarks/{network}`)
- USD and ETH reference rates for comparison context
- Historical shape: `{ data: [{ apy: { "7day": { base, reward, total } }, timestamp }] }`

**Historical vault data** (`/v2/historical/{network}/{vaultAddress}/apy`)
- Used for APY history charts on individual vault pages (future)

## Page Template Structure

Every research page follows this structure:

```
1. Hero
   - H1: keyword-rich title (e.g. "Best USDC Yields in DeFi")
   - Current best APY (largest number on the page)
   - "Last updated: [date]" + "Data: vaults.fyi"

2. Live data table (top 10 vaults)
   - Columns: Protocol | Vault | Network | 7-day APY | TVL | Risk
   - Sorted by 7-day APY descending
   - Risk derived from reputationScore

3. APY benchmark chart
   - 30/90-day USD or ETH benchmark trend
   - Simple Chart.js line chart, rendered client-side
   - Shows "market context" — is yield high or low vs. history?

4. Explanatory section
   - "Where does this yield come from?" — templated prose per asset/protocol type
   - "What are the risks?" — pulled from vault flags + general protocol risk notes

5. FAQ (3–5 questions)
   - Rendered as visible Q&A on page
   - Also encoded as JSON-LD FAQPage schema
   - These are the primary AEO surface — AI systems extract and cite these directly

6. Internal links
   - Related pages (e.g. USDC page links to stablecoin comparison, Morpho page)
   - "Explore all vaults" → app.vaults.fyi

7. Structured data (in <head>)
   - JSON-LD: FAQPage, Dataset, BreadcrumbList
```

## SEO / AEO Specifics

### For traditional SEO
- Keyword-rich `<title>` and `<meta description>` per page
- `<h1>` matches primary query
- Sitemap.xml auto-generated by Astro
- `robots.txt` allowing full crawl
- Canonical URLs
- Fast load times (static HTML, minimal JS)
- `last-modified` in HTTP headers (Vercel does this automatically)

### For AEO (AI citation)
- FAQPage JSON-LD on every page — the primary signal for AI answer extraction
- Dataset JSON-LD for structured data tables
- Clear attribution: "Data sourced from vaults.fyi, updated daily"
- Unambiguous factual statements (not hedged or vague)
- Short, direct answers before detailed explanation (AI systems prefer lead-with-answer structure)
- Internal links to primary domain (vaults.fyi) reinforce authority

## Hosting Decision

### Pilot: GitHub Pages

GitHub Pages works for a daily-regenerated static site. The CTO's suggestion is reasonable for getting started fast. Constraints to be aware of:
- No server-side logic (fine for static)
- CDN is basic compared to alternatives
- Custom domain requires DNS config
- No automatic HTTPS redirect for custom subdomains without workaround

**Verdict: acceptable for pilot.**

### Recommended upgrade: Vercel

Once the pilot is validated, migrate to Vercel. Reasons:

| Factor | GitHub Pages | Vercel |
|---|---|---|
| CDN | Basic GitHub CDN | Global edge network (100+ PoPs) |
| Performance headers | Manual | Automatic (Cache-Control, ETag) |
| Deploy triggers | GitHub Actions push | Native GitHub integration, no extra config |
| Custom domains | Supported | Supported + easier SSL |
| ISR / future live data | Not possible | Possible (add server routes later) |
| Cost | Free | Free (Hobby tier covers this) |

Vercel is the same workflow as GitHub Pages (push to deploy) but strictly better in every dimension. No meaningful reason to stay on GitHub Pages once the pilot proves out.

## Domain / Hosting Decision

### Pilot: `blog.vaults.fyi/research/*`

- Fastest to launch (path routing via Vercel, no DNS changes)
- Inherits blog.vaults.fyi domain authority
- Ghost handles `/` and `/research` handles programmatic pages via separate deployment

### Recommended long-term: `vaults.fyi/research/*`

This is the right call if the main app team can accommodate it. Here's why:

**Root domain authority compounds.** Pages at `vaults.fyi/research/best-usdc-yields` benefit from all backlinks to `vaults.fyi` — the app, the API docs, press coverage, partner integrations. A subdomain (`blog.vaults.fyi`) is treated as a separate site by Google and must build authority independently.

**AI citation preference.** LLMs and AI search tools weight canonical domain authority similarly to Google. Content at `vaults.fyi/research/*` is more likely to be cited than content at `blog.vaults.fyi/research/*` because the root domain has stronger signals.

**Practical path:** for the pilot, `blog.vaults.fyi/research/*` is fine. Once pages demonstrate ranking and traffic, make the case to the app team to serve `/research/*` as a static path on the main domain via reverse proxy or Vercel routing.

## SDK Reference

### TypeScript SDK (`@vaultsfyi/sdk`)

```typescript
import { VaultsSdk } from '@vaultsfyi/sdk'

const sdk = new VaultsSdk({ apiKey: process.env.VAULTS_API_KEY })

// Get top USDC vaults
const vaults = await sdk.getDetailedVaults({
  query: { assets: ['USDC'], perPage: 50, page: 0 }
})

// Get benchmark history
const benchmarks = await sdk.getHistoricalBenchmarks({
  path: { network: 'ethereum' },
  query: { from: '2025-01-01', to: '2025-04-15' }
})
```

### Python SDK (`vaultsfyi`)

```python
from vaultsfyi import VaultsSdk

client = VaultsSdk(api_key=os.environ["VAULTS_API_KEY"])

vaults = client.get_detailed_vaults(assets=["USDC"], per_page=50)
benchmarks = client.get_historical_benchmarks(network="ethereum")
```

### Key data notes

- APY fields (`apy`, `apy7d`, `apy30d`) are raw decimals — multiply by 100 for display
- Use `apy7d` for displayed rates on research pages (smoother, more representative)
- Benchmark historical shape: `data[].apy["7day"].total` (not `data[].apy.total`)
- Pagination: `page` is zero-indexed; response includes `nextPage: null` on last page
